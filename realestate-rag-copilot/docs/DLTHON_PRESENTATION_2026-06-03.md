## Slide 2. 문제 정의 — 왜 전세사기는 줄지 않는가 (60초)

**슬라이드 내용**
- 2023~2025년 전세사기 피해자 누적 약 27,000명
- 평균 보증금 1.5억 — 인생 자산이 한순간에 사라짐
- 사용자(특히 초보 임차인)가 확인해야 할 것: **8가지 이상**
  - 등기부등본 / 건축물대장 / 실거래가 / 전세가율 / 선순위 채권 / 임대인 신뢰도 / 보증보험 가능 / 위반건축물
- 정보는 모두 **분산**되어 있음 (등기소, 정부 API, 상업 DB, 부동산 카페)
- 공인중개사·법률 전문가에게 매번 묻기엔 비용·시간 부담

**스피커 노트**
> "전세사기 피해는 여전히 매년 수천 건씩 발생합니다. 문제는 정보의 양이 아니라 **분산**입니다. 등기부등본은 인터넷등기소에서, 실거래가는 국토부에서, 건축물대장은 건축HUB에서… 사용자가 8가지 사이트를 돌아다녀야 합니다. 게다가 각 정보를 어떻게 해석해야 하는지조차 막막합니다."

## Slide 3. 단순 RAG의 한계 (60초)

**슬라이드 내용**

| 단순 RAG가 잘하는 것 | 단순 RAG가 못하는 것 |
|---|---|
| ✓ "전세 계약 시 확인할 것?" 체크리스트 검색 | ✗ "이 집 등기부에 근저당이 있는가?" |
| ✓ 법률 조항·가이드 인용 | ✗ "현재 시세 대비 보증금이 적정한가?" |
| ✓ 일반적 안내문 생성 | ✗ "임대인이 최근 압류된 적이 있는가?" |
| | ✗ "이 주소가 위반건축물인가?" |

→ **체크리스트는 알지만 실제 데이터는 모름**
→ LLM이 단정하면 법적 책임 발생 ("이 계약은 안전합니다" 같은 표현)

**스피커 노트**
> "RAG는 '뭘 확인해야 하지?'에는 잘 답합니다. 하지만 '이 집은 어떻지?'에는 답할 수 없습니다. 답하려면 실시간 외부 데이터가 필요하고, 그 데이터를 해석하는 룰이 필요하며, 단정 표현을 막는 안전장치가 필요합니다. 그래서 RAG 위에 Agent를 얹기로 했습니다."

## Slide 4. 트러스트 아크의 접근 — RAG + Agents 샌드위치 (60초)

**슬라이드 내용** (도식)

```
┌─────────────────── 사용자 입력 ───────────────────┐
│ 주소 · 보증금 · 질문                              │
└───────────────────────┬────────────────────────────┘
                        │
          ┌─────────────▼─────────────┐
          │ [LLM] Planner Agent        │  ← 의도 분류 + 실행 계획
          └─────────────┬─────────────┘
                        │
   ┌────────────────────▼────────────────────┐
   │ 결정론적 외부 데이터 수집 파이프라인       │
   │  ├─ Location Context (VWorld/네이버)     │
   │  ├─ Market Data (data.go.kr 실거래가)    │
   │  ├─ Building Register (건축HUB)         │
   │  ├─ Registry (CODEF 등기부등본)         │
   │  └─ Search Context (네이버 검색)        │
   ├─ RAG Evidence (jeonse_risk_checklist)   │
   └─ Risk Scoring (룰 기반 + 보수적 floor)  │
                        │
          ┌─────────────▼─────────────┐
          │ [LLM] Summarizer Agent     │  ← 자연어 요약 + 액션 순위
          └─────────────┬─────────────┘
                        │
              ┌─────────▼─────────┐
              │ Validation Agent  │  ← 단정 표현 검증
              └─────────┬─────────┘
                        ▼
                  사용자 화면
```

**스피커 노트**
> "구조는 샌드위치입니다. 위에 LLM Planner가 사용자 질문을 해석해 '뭘 가장 중요하게 봐야 할지' 결정합니다. 그 결정에 따라 결정론적인 외부 API 파이프라인이 돕니다. 데이터가 모이면 다시 LLM Summarizer가 자연어로 요약하고, 마지막에 Validation Agent가 단정 표현을 차단합니다. **결정은 LLM이, 사실 수집은 결정론이** — 환각과 신뢰성을 같이 잡았습니다."

## Slide 5. 시스템 아키텍처 (90초)

**슬라이드 내용** (기술 스택 + 핵심 메커니즘)

**기술 스택**
- Next.js 16 + TypeScript (단일 런타임, Vercel 배포)
- OpenAI gpt-4o-mini (Planner) + gpt-4o (Summarizer)
- 외부 API: VWorld, 네이버 지도/검색, data.go.kr, 건축HUB, CODEF

**11개 Agent 구성**

| 위치 | Agent | 역할 |
|---|---|---|
| 입력 LLM | **Planner** | 의도 분류 + 실행 계획 |
| 데이터 | Location Context | 지오코딩 |
| 데이터 | Market Data | 전월세·매매 실거래가 |
| 데이터 | Building Register | 건축물대장 |
| 데이터 | Registry | 등기부등본 (자동 모드: 안내만) |
| 데이터 | Search Context | 네이버 검색 |
| 지식 | RAG Evidence | 체크리스트 근거 |
| 계산 | Risk Scoring | 점수 + 보수적 floor |
| 출력 LLM | **Summarizer** | 자연어 요약 + 액션 순위 |
| 검증 | Validation | 단정 표현 차단 |
| 보고 | Report | 리포트 조립 |

**핵심 메커니즘 3가지**
1. **Mock 누수 차단** — 외부 API 실패 시 mock 숫자로 덮지 않고 명시적 "missing" 표시
2. **점수 분해 (Score Breakdown)** — `시세 기반 X점 + 데이터 품질 보정 Y점 = 최종`
3. **Agent Trace** — 모든 호출이 시간순으로 기록되어 검증 가능

**스피커 노트**
> "Agent는 총 11개입니다. 양쪽 끝의 두 개는 LLM이 결정을 내리는 자율 부분, 가운데는 결정론적 데이터 수집입니다. 특히 세 가지 메커니즘이 차별점입니다 — 첫째 mock 누수 차단으로 정직한 화면, 둘째 점수 분해로 '왜 이 점수인지' 설명 가능, 셋째 Agent Trace로 모든 LLM 결정을 사용자가 검증할 수 있습니다."

## Slide 7. 시연 (2) Planner의 의도 해석 + 실행 계획 (75초)

**슬라이드 내용** (Planner 카드 스크린샷)

> "사용자가 알고 싶어 하는 것: 이 집 전세 계약 전에 무엇을 확인해야 하나요?"

**Intent Tags**: `#전세보증금 반환보증` `#깡통전세/전세사기` `#권리관계 변동`

**Emphasis**:
- 권리관계 우선 확인
- 전세보증금 반환보증 가입 가능성

**AI가 결정한 실행 계획**:
- 🔴 **최우선** Registry (등기부등본) — 사용자가 권리 확인 우려를 표현
- ⚪ 기본 Market Data (실거래가) — 시세 검증은 보조
- ⚪ 기본 Building Register (건축물대장)
- 🟫 건너뜀 Search Context (외부 검색) — 직접 관련 없음

**스피커 노트**
> "막연한 질문이지만 Planner는 gpt-4o-mini로 의도를 분류합니다. '확인 사항' 같은 일반적 질문에서도 권리관계가 가장 큰 우려라고 판단했고, 그에 맞춰 등기부등본을 **최우선(critical)** 으로 지정합니다. 외부 검색 같은 부차적 단계는 **건너뜀(optional)** 으로 분류해 시간·비용도 절약합니다. **이게 단순 RAG와 가장 다른 점입니다 — 사용자별로 동작이 바뀝니다.**"

## Slide 8. 시연 (3) 외부 데이터 수집 + 점수 분해 (90초)

**슬라이드 내용** (분석 결과 화면 스크린샷)

```
📌 데이터 출처 상태 (7개 카드, 모두 명시)
   ✓ 지도 지오코딩      ✓ 법정동코드
   ✓ 전월세 실거래가    ✓ 매매 실거래가
   ✓ 건축물대장        ⚠ 등기부등본 (수동 분석)
   ✗ 외부 검색 (skip)
```

**검토 필요 / 60점**

```
점수 분해
████████████████████░░░░░░░░░░░░░░░░░░░░░░  60/100
■ 52점 · 전세가율 24% · 안전 구간 (시세 기반)
□ +8점 · 등기·보증보험 미확인 보정 (데이터 품질)
─────────────────────
  최종 60점 · 검토 필요
```

**스피커 노트**
> "데이터 수집 결과입니다. 7개 외부 API 상태가 카드로 표시되고, mock 숫자 없이 **실제로 무엇이 확보됐는지** 정직하게 보여줍니다. 점수 분해가 핵심인데 — 전세가율은 24%로 시세상 안전 구간(52점)이지만, 등기·보증보험 미확인 페널티가 +8점 추가돼 결국 60점 '검토 필요'가 됩니다. **'안전한데 왜 검토 필요?' 같은 인지 부조화를 누적 막대 하나로 설명합니다.** 게다가 Planner가 critical로 찍은 등기부등본이 미확보된 상태이므로, 사용자에게 명시적 경고가 추가됩니다."

## Slide 9. 시연 (4) Summarizer의 자연어 출력 (60초)

**슬라이드 내용** (summary + next_actions)

**요약 (Summarizer가 매번 새로 씀)**:
> "현재 데이터 기준으로 전세가율은 24%로 시세 측면 위험은 높지 않지만, 등기부등본 권리관계와 전세보증금 반환보증 가입 가능성이 아직 확인되지 않았습니다. 사용자가 우려하시는 권리관계를 먼저 확인하시는 것을 권장합니다."

**다음 확인 액션 (우선순위 매겨짐)**:
1. 등기부등본을 발급해 근저당권·압류·가압류 여부 확인 ← **최우선**
2. 전세보증금 반환보증 가입 가능 여부와 한도 확인
3. 임대인의 체납·신용 이력 확인
4. 특약에 잔금 전 권리변동 금지 조항 포함
5. 공인중개사·법률 전문가 검토

**스피커 노트**
> "마지막으로 Summarizer(gpt-4o)가 모든 데이터와 Planner의 의도를 받아 사용자별 자연어 요약을 만듭니다. 같은 분석 결과여도 '권리관계 우려' 사용자에게는 등기부 확인이 1번 액션이 되고, '시세 우려' 사용자에게는 실거래가 비교가 1번 액션이 됩니다. 단정 표현은 Validation Agent가 마지막에 차단합니다."

### 디엘톤 심사위원 관점에서 강조할 포인트

1. **"RAG 주제를 어떻게 확장했는가?"** — 슬라이드 3·4에서 RAG의 한계를 명확히 짚고, Agent로 어떻게 보완했는지 보여줄 것
2. **"실제로 동작하는가?"** — 슬라이드 6~9의 시연이 핵심. 가능하면 **라이브 데모**, 안 되면 스크린샷 시퀀스
3. **"기술적 깊이는 있는가?"** — 슬라이드 5의 아키텍처가 평가 포인트. 11개 Agent, mock 누수 차단, 점수 분해, Trace 검증 — 4가지 메커니즘 모두 언급
4. **"차별점이 무엇인가?"** — 슬라이드 10의 비교 표가 한눈에 정리됨

### 시간 분배 주의
- 슬라이드 1~5(설명)는 합쳐서 4분 30초가 한계. 길어지면 시연 시간이 부족해짐.
- 시연 슬라이드 6~9는 4분 30초가 충분. 라이브 데모 시 페이지 로딩 5초 정도는 양해 표현 ("외부 API를 5개 호출하느라 잠시 기다려주세요").
- 슬라이드 10은 1분으로 압축. 비교 표는 자세히 안 읽고 핵심만 짚을 것.

### 예상 질문 대비
1. **"LLM 비용은 얼마인가?"** → 분석당 약 $0.013 (gpt-4o-mini + gpt-4o). 월 1000건 처리 시 ~$13.
2. **"환각이 일어나면 어떻게 되는가?"** → 핵심 수치(시세·점수)는 결정론적 코드에서 산출. LLM은 의도 분류와 자연어 정리만 담당. 단정 표현은 Validation Agent가 차단.
3. **"등기부등본 자동 열람은 왜 안 하는가?"** → 수수료 발생 + 사용자 인증값 필요. 자동 분석에서는 안내만 하고 사용자가 명시적으로 실행하도록 분리.
4. **"확장 가능한가?"** — Next.js 단일 런타임 + Vercel. agent를 추가하려면 새 함수만 만들고 plan에 등록하면 됨.
5. **"왜 OpenAI인가? Claude는?"** — Vercel 환경변수에 이미 OpenAI 키가 있었음. SDK 추상화는 어렵지 않아 향후 멀티프로바이더 지원 가능.

### 데모 시 주의사항
- 라이브 호출이 30초 이상 걸리면 미리 캡쳐한 화면으로 대체
- 시연 입력은 미리 정해진 케이스 사용 (마포 망원로 / 1.1억) — 즉석 입력하면 API 실패 시 대처 어려움
- 화면 공유 해상도가 낮으면 점수 분해 막대가 안 보일 수 있음 → 미리 확대된 스크린샷 준비
