# 창업 인허가·규제 검토 도메인 기획서

**작성일**: 2026-06-04
**전제**: 트러스트 아크의 부동산 분석 인프라(11개 agent + RAG + LLM 샌드위치 + 점수분해 + Trace)를 인접 도메인으로 확장
**범위**: 음식점·미용실·학원·PC방·노래방 등 주요 5+ 업종의 위치별 인허가·규제 적합성 자동 검토

---

## 1. 왜 이 도메인인가

### 1.1 부동산 도메인의 한계 인식
현재 트러스트 아크는 "전월세 위험 검토"에 머물러 있고, 사용자가 진짜 필요로 하는 정보 분석(금융 데이터·임대인 신뢰도·시나리오 시뮬레이션)은 추가 인프라 부담이 큼. 마이데이터 사업자 등록(자본금 5억+) 같은 진입 장벽이 단기 발전을 막음.

### 1.2 창업 인허가 도메인의 매력
| 항목 | 부동산 (현재) | 창업 인허가 (제안) |
|---|---|---|
| 핵심 데이터 출처 | 공공 API 5종 | **공공 API 10종+ (전부 무료)** |
| 사용자 입력 부담 | 주소·계약조건 | 주소·업종·운영조건 |
| 마이데이터 필요 | 깊은 분석 필수 | **불필요** |
| 법적 부담 | 권리관계 자문 회색지대 | **정보 제공형, 회색지대 없음** |
| 트러스트 아크 인프라 재사용 | - | **85%** (좌표·건축물대장·법정동코드·Planner·점수분해·RAG 전부 그대로) |
| 시장 크기 | 임차·매수자 (수백만) | 연 신규 사업자 100만+ + 기존 소상공인 280만 |

### 1.3 사용자 통증
- 카페·식당·공방 창업 직전 "**이 위치에서 가능한가?**" 확인 부담 큼
- 공인중개사도 정확히 답 못함 (영업제한·자치구 조례까지 확인하지 못함)
- 위반 시 영업정지 또는 폐업 위험
- 현재 시장: 행정사·법무사 의뢰 시 건당 30~50만원, 자체 확인 시 평균 2~3주 소요

## 2. 현재 트러스트 아크 자산 매핑

### 2.1 그대로 재사용되는 자산 (수정 거의 없음)
| 자산 | 부동산에서의 용도 | 창업 인허가에서의 용도 |
|---|---|---|
| VWorld·네이버 지오코딩 | 매물 좌표 | 점포 좌표 |
| 법정동코드 (LAWD_CD) | 실거래가 조회 | 관할 자치구 매칭 |
| 건축HUB 건축물대장 (주용도) | 다세대주택 여부 확인 | **2종 근린생활시설 여부 확인** (핵심) |
| 위반건축물 표시 | 임차 위험 신호 | **영업 불가 사유** (필수) |
| Planner Agent (gpt-4o-mini) | 사용자 의도 분류 | **업종 분류 + 우선순위 결정** |
| Summarizer Agent (gpt-4o) | 자연어 요약 | **인허가 결과 자연어 안내** |
| Score Breakdown 누적 막대 | 시세+데이터 품질 분해 | **체크 항목별 분해** (용도지역+학교거리+위반건축물) |
| Data Statuses 카드 | 외부 데이터 출처 상태 | **각 검증 단계 통과 여부** |
| Agent Trace | 결정 과정 검증 | 동일 |
| Validation Agent | 단정 표현 차단 | 동일 ("영업 가능합니다" 같은 단정 금지) |
| Execution Plan (B 옵션) | critical/normal/optional 분기 | **업종별 critical 단계가 달라짐** (음식점: 정화조 critical, PC방: 학교거리 critical) |

### 2.2 새로 필요한 자산
- 토지이용계획 API 연동 (용도지역 판정)
- LOCALDATA 연동 (동종업종 밀집·이격거리)
- 학교알리미 + 청소년시설 데이터 (정화구역 검증)
- 자치법규(ELIS) RAG (자치구별 영업제한 거리·시간)
- 식품위생법·학원법·공중위생관리법·게임산업진흥법 RAG
- 업종별 시설기준 매핑 테이블 (룰 코드)

## 4. 즉시 신청 추천 TOP 10 API

거의 모두 **공공데이터포털(data.go.kr) 키 하나로 신청 가능**. 트러스트 아크는 이미 키 보유 중 → 활용 신청만 추가하면 됨.

| 순위 | API | 즉시? | 핵심 가치 | 트러스트 아크 매핑 |
|---|---|---|---|---|
| ① | **LOCALDATA** (행정안전부) | ✅ 즉시 | 전국 1,700+ 자치단체 인허가 사업장 전수 (200+ 업종 좌표) | 동종업종 거리 / 상권 밀집도 |
| ② | **국토부 LURIS** (토지이용계획) | 1~3일 | 용도지역·지구 | 인허가 가능 출발점 |
| ③ | **법제처 국가법령정보** | ✅ 즉시 | 식품위생법·학원법·공중위생법 등 본문 | RAG 핵심 |
| ④ | **학교알리미** | 1~2일 | 학교 좌표·학생수 | 정화구역 200m 검증 |
| ⑤ | **자치법규(ELIS)** | 1~3일 | 자치구 조례 본문 | 자치구별 차등 규제 RAG |
| ⑥ | **소상공인진흥공단 상권정보** | 1~2일 | 업종별 점포수·매출 추이 | 시장 시뮬레이션 |
| ⑦ | **국가공간정보(NSDI)** | ✅ 즉시 | 용도지구·도시계획시설 WMS/WFS | 공간 규제 보강 |
| ⑧ | **심평원 의료기관** | 1~2일 | 약국·의원 좌표 | 일부 업종 이격거리 |
| ⑨ | **여성가족부 청소년시설** | 1~2일 | 청소년시설 좌표 | 청소년 보호 거리 |
| ⑩ | **국세청 사업자등록 진위확인** | 1~2일 | 사업자번호 진위 + 휴폐업 | 사업주 검증 + 임대인 법인 검증 |

### 신청 실행 순서
**Day 1**: ①·③·⑦ (즉시 발급) + ②④⑤⑥⑧⑨⑩ 활용 신청 일괄 제출
**Day 2~3**: 승인 받은 API들 sample call 테스트, 응답 스키마 정리
**Day 4~**: ① + ②를 가장 먼저 연결 (MVP 핵심 가치)

## 5. 새 에이전트 구성

기존 11개 agent에 새 agent **6~8개**를 추가. 모두 **이미 작성된 `data-go-kr.ts` 헬퍼** 위에 구축.

### 5.1 기존 11개 그대로 사용
- **Location Context Agent** — 점포 좌표·법정동코드
- **Building Register Agent** — 주용도·위반건축물 (확장: 정화조·주차장)
- **RAG Evidence Agent** — 법령·조례 RAG (확장)
- **Planner Agent** — 업종 분류 + execution plan
- **Summarizer Agent** — 자연어 결과 안내
- **Risk Scoring Agent** — 적합 점수 산출
- **Validation Agent** — 단정 표현 차단
- **Report Agent** — 리포트 조립
- **Search Context Agent** — 외부 맥락 (선택)
- **Market Data Agent** — 비활성화 또는 상권 데이터로 재용도
- **Registry Agent** — 비활성화 또는 사업자등록 진위로 재용도

### 5.2 신규 Agent 6개
| Agent | 입력 | 출력 | 사용 API |
|---|---|---|---|
| **Zoning Agent** | 좌표 | 용도지역(주거/상업/공업/녹지) + 가능 업종 | LURIS (②) |
| **Building Suitability Agent** | 건축물대장 주용도 + 업종 | "근린생활시설 적합/부적합" + 사유 | 건축HUB (기존) |
| **School Zone Agent** | 좌표 | 반경 50/200m 내 학교 N개 + 정화구역 판정 | 학교알리미 (④) |
| **Youth Protection Agent** | 좌표 + 업종 | 청소년시설 인근 + 영업제한 적용 여부 | 청소년시설 (⑨) |
| **Competition Density Agent** | 좌표 + 업종 | 반경 500m 동종업종 N개 + 이격거리 통과/실패 | LOCALDATA (①) |
| **License Requirement Agent** | 업종 + 자치구 | 필요 인허가·수수료·처리기간·결격사유 | 법령 RAG (③) + 조례 RAG (⑤) |

### 5.3 Planner의 execution_plan 확장
Planner는 업종을 보고 **critical/normal/optional**을 다르게 결정:

```
[일반음식점]
  Zoning            : critical (2종 근린생활시설 가능 여부)
  Building          : critical (정화조 용량)
  School Zone       : normal (직접 영향 적음, 단 청소년 출입 시간)
  Youth Protection  : optional
  Competition       : normal (참고)
  License           : critical (위생교육·시설기준)

[PC방]
  Zoning            : critical
  Building          : normal
  School Zone       : critical (200m 정화구역)
  Youth Protection  : critical (청소년 출입제한 + 시간)
  Competition       : optional
  License           : critical

[학원]
  Zoning            : critical
  Building          : critical (학원법 면적·층수)
  School Zone       : normal
  Youth Protection  : optional
  Competition       : normal
  License           : critical (학원설립등록)
```

### 5.4 점수 분해 활용
```
음식점 영업 가능 점수 88/100 =
  용도지역 적합 (+30)
  + 건축물 주용도 적합 (+25)
  + 정화조 용량 충분 (+15)
  + 학교 200m 외 (+10)
  + 동종업종 이격 적합 (+8)
  - 위반건축물 (없음)
  + 추가 인허가 절차 (1단계 보건소만)
```

## 7. 법적·운영 고려사항

### 7.1 법적 안전장치
- "**정보 제공형**"으로 포지셔닝. "법무 자문이 아닌 참고 정보" 명시.
- Validation Agent를 통해 단정 표현 차단 (예: "영업 가능합니다" → "현재 데이터 기준 영업 가능성 높음, 최종 확인 필수")
- 조례·법령은 자주 개정되므로 RAG 출처에 **마지막 갱신일** 명시
- 책임 한계 문구를 모든 결과에 노출

### 7.2 데이터 갱신 주기
- LOCALDATA: 일별 갱신 (자동)
- LURIS·NSDI: 월~분기별
- 법령 (법제처): 개정 시 갱신
- 자치법규(ELIS): 개정 시 갱신
- 학교알리미: 학기별

→ **법령·조례 RAG는 월 1회 자동 재인덱싱** 필요. 코드 작성 시 cron job 또는 GitHub Actions로.

### 7.3 사용자 개인정보
- 사업자등록번호 진위 확인은 사용자 본인 정보만 처리
- 점포 주소는 분석 후 즉시 폐기 또는 익명화 저장
- 결격사유 자가확인은 클라이언트 측에서만 처리 (서버 저장 X)

### 7.4 운영 비용 추정
- 공공 API: 전부 무료 (개발/일반 사용 트래픽 한도 내)
- OpenAI (Planner gpt-4o-mini + Summarizer gpt-4o): 분석당 약 $0.013
- 인프라(Vercel): 트래픽 따라 변동, 월 5만원 이하 (Hobby 한도 내)
- **월 운영비**: 분석 1,000건당 ~$13 + Vercel ~$5 = **약 2~3만원**

## 8. 이미 작성된 인프라

### 8.1 `data-go-kr.ts` 공통 헬퍼 (2026-06-04 작성)
모든 공공데이터포털 API 호출의 기반. 새 agent들은 이 위에 30~80줄 짜리 wrapper만 추가하면 됨.

```ts
import { callDataGoKrJson, summarizeAttempt } from "@/lib/server/data-go-kr";

// 새 Agent 예: School Zone Agent
const result = await callDataGoKrJson<SchoolApiResponse>({
  endpointName: "학교알리미",
  url: "https://...../OpenApi3/SchoolBasicInfo",
  params: { Type: "json", pSize: 100, ATPT_OFCDC_SC_CODE: "B10" }
});
if (!result.ok) {
  trace.record("School Zone Agent", "querySchools", inputSummary, summarizeAttempt(result.attempt), "failed");
  return null;
}
return filterByRadius(result.data, lat, lng, 200);
```

**헬퍼가 자동 처리하는 것**: 키 형태(encoded/decoded) 선택 · URL 안전 인코딩 · timeout · 로그 마스킹 · 표준 envelope (`resultCode`/`totalCount`/`items.item[]`) 추출 · 진단 객체

### 8.2 기존 트러스트 아크 코드 그대로 사용
- `lib/server/agent-runtime/trace.ts` — TraceAgent union에 새 6개 agent 추가만 하면 됨
- `lib/types.ts` — AnalyzeResponse 구조 그대로
- `app/api/analyze/route.ts` — orchestration 패턴 그대로
- `components/RiskReport.tsx` — PlannerInsightPanel·ScoreBreakdownPanel·AgentReportPanel 모두 재사용

### 8.3 RAG 인프라 확장 방향
현재 `rag_docs/jeonse_risk_checklist.md` 1개 파일을 4계층 RAG로:
1. **법령** — 식품위생법·학원법·공중위생법·게임산업법 (D1~D5)
2. **조례** — 서울 25개구 영업제한 조례 (E1)
3. **시설기준 매핑** — 업종별 면적·환기·분리 등 (C4, JSON 룰)
4. **인허가 절차** — 단계·서류·수수료·기간 (G5·G6)

벡터 임베딩 또는 LLM 직접 입력 모두 가능 (분량 적음).

## 10. 부동산 도메인과의 관계

이 새 도메인은 **부동산 도메인을 대체하는 것이 아니라 병행** 가능:

- **공통 코어**: 좌표·법정동코드·건축물대장·Planner·Summarizer·Risk Scoring·Validation
- **분기 지점**: `/api/analyze`(부동산) vs `/api/business-permit`(창업)
- **공유 자원**: data-go-kr helper, agent runtime, trace, score breakdown

향후 사용자 입장에서는 같은 점포·주소를 두고 두 가지 관점으로 볼 수 있음:
- 부동산: "이 매물 임차/매수 안전한가?"
- 창업: "이 매물에서 음식점 창업 가능한가?"

**서비스 이름·브랜딩만 확장**하면 됨 (예: "트러스트 아크 — 부동산·창업 검토 코파일럿").
